    Building tool-to-module: A Step-by-Step Claude Code Tutorial

    A worked tutorial for building an LLM-assisted CLI tool that converts bioinformatics scripts and Git repositories into submission-ready nf-core modules. Built with Claude Code, prompt by prompt, with real-world testing against Celltypist.

    Overview

    Contributing a module to nf-core means writing main.nf, meta.yml, environment.yml, and a working nf-test — then getting them through lint and review. For well-known tools with a Bioconda package, nf-core modules create gives you a starting point. For everything else — custom scripts, in-house tools, novel software not yet in Bioconda — you start from a blank file and figure it out by reading existing modules.

    tool-to-module automates the blank-file problem for that second category. Point it at a script or Git repository, and it generates a complete nf-core module directory: correct channel structure, BioContainers lookup, nf-test with real or derived test data, and a review report flagging anything that needs human attention before submission. It is aimed at bioinformaticians who know their tool well but find the nf-core module format fiddly to get right the first time.

    A note on what "automated" means here. The tool is LLM-assisted, not fully autonomous. Claude handles the inference step — reading source code and docs to determine channel names, types, and the shell command — but the pipeline around it is deterministic: discovery is rule-based, container resolution queries real APIs, and generation uses fixed Jinja2 templates. The output is a best-effort module that the author should review before submitting. It eliminates the blank-page problem and handles the mechanical work; it does not replace domain judgement. The Celltypist example in the manual checkpoint (Prompt 16) is a realistic illustration of what "best effort" looks like: correct container, correct channel structure, real command in the script block — but output globs that may need splitting and optional flags that need human verification.

    How it relates to existing tooling. tool-to-module is designed to sit alongside the tools the nf-core community already uses, not replace them:

    • nf-core modules create scaffolds the file structure. tool-to-module fills in the content — it is the step after the scaffold.
    • A Claude Code skill (a markdown instruction file) can handle simple, well-known tools reasonably well, but cannot enforce structural invariants programmatically, call BioContainers and bio.tools APIs consistently, or run the validate-module suite. The right mental model is skill plus tool-to-module: the skill provides a natural language interface, the tool provides the reliability underneath.
    • Seqera AI works at the pipeline level — generating full Nextflow pipelines, converting scripts, debugging runs. The two tools complement each other naturally: Seqera AI to assemble the pipeline, tool-to-module to contribute individual modules back to the community. Seqera AI also exposes a describe_nfcore_module MCP tool that returns structured module metadata, which could power the --existing-modules flag and the regression test suite once the core tool is stable.

    Who Is This For?

    tool-to-module is useful in three distinct situations, each with a different primary concern:

    Academic labs — A PhD student or postdoc has a working analysis script that a collaborator or reviewer wants shared as a reusable nf-core module. The barrier is usually not the biology — it's learning DSL2 syntax, nf-test conventions, container resolution, and the meta.yml schema all at once. tool-to-module convert handles all of that from a single script file. The student runs it, reviews the output, fills in the TODO stubs (if any), and has a submission-ready module in an afternoon rather than a week.

    Industry (pharma/biotech/CROs) — A computational biology team maintains a collection of internal pipeline scripts and wants to standardise them against nf-core conventions for reproducibility audits or community contribution. The --no-interaction flag and the Python API (Prompt 19) support CI/CD integration: the tool can be called programmatically to convert a directory of scripts as part of a build pipeline, producing a structured report of what succeeded, what hit Tier 5 (too complex for auto-generation), and what needs human review. The --all-functionalities flag handles multi-entrypoint repos without human prompting.

    nf-core community contributors — Someone writing a new module from scratch can use tool-to-module to generate the boilerplate from a prototype script, then refine it. Maintainers reviewing external PRs can run tool-to-module review against the submitted module and paste the output directly into review comments rather than writing them by hand. The test and fix commands are also useful for bringing existing modules up to date when nf-core conventions change.

    A note on scope. The tool is deliberately not a replacement for understanding nf-core conventions — the generated output always needs a human review pass before submission. Its value is eliminating the mechanical work so that review energy goes into the biology and design decisions rather than YAML formatting and lint errors.

    CLAUDE.md

    This is the most important file — it gives Claude Code persistent context about your project conventions, nf-core standards, and how to handle Nextflow-specific patterns.

    # CLAUDE.md — tool-to-module

## Build Order
Build prompts in document order (1 through 31). Two prompts write tests before
their implementation prompt: Prompt 5 (standards contract) and Prompt 12
(container contract) — do not skip ahead when their tests are red from ImportError
or AttributeError. That is expected and intentional: the tests define the contract,
the next prompt satisfies it.
Do not proceed to the next prompt until all tests for the current prompt are green.
Full prompt text is in docs/development-tutorial.md.

## Project Purpose
CLI tool that converts scripts/Git repos into submission-ready nf-core modules.
Uses LLM inference to determine process inputs, outputs, containers, and parameters.

## Architecture
- `ingest.py` — accepts local file paths, directories, or Git URLs; returns CodeSource with repo manifest
  Also accepts optional --docs list of URLs/paths; fetches content into CodeSource.doc_sources
  Also accepts optional --existing-modules list of module directory paths; parses into
  CodeSource.existing_modules for use by discover, infer, and generate
- `discover.py` — detects distinct functionalities using rules + LLM; interactive selection
  Rule-based detectors use code only; LLM fallback incorporates doc_sources content when present
- `assess.py` — assigns Tier 1-5 complexity per FunctionalitySpec (not per CodeSource)
- `infer.py` — calls Anthropic API scoped to one FunctionalitySpec's code section
- `container.py` — two-phase container handling: discover all options, then select one
- `bioconda.py` — check Bioconda for existing packages; generate meta.yaml scaffold for new submissions
- `standards/` — extractable subpackage; no imports from rest of tool_to_module
  - `loader.py` — Standards class + get_standards() singleton
  - `data/nf_core_standards.json` — versioned schema; `data/STANDARDS_CHANGELOG.md`
  - Import as: `from tool_to_module.standards import get_standards`
- `test_data_match.py` — Strategy 1: match channel specs to existing nf-core/test-datasets files
- `test_data_derive.py` — Strategy 2–4: derive test data from existing nf-core files,
  chain an upstream nf-core module in an nf-test setup block, or fall back to Nextflow stub mode
- `test_gen.py` — orchestrates test data strategy and produces TestSpec + derive_test_data.sh
- `generate.py` — renders Jinja2 templates into module files
- `quick_lint.py` — fast post-generate structural checks (no subprocess); feeds warnings to CLI output
- `validate.py` — runs nf-core lint + nf-test, captures output, returns structured TestReport
- `fix.py` — classifies failures, proposes rule-based or LLM-assisted fixes as diffs
- `review.py` — static analysis against nf-core style conventions, returns ReviewReport
- `validate_cli.py` — Click group for validate-module entry point (test/fix/review commands)
- `api.py` — clean public API for programmatic use by other tools; no Rich output

## Entry Points
Two CLI entry points ship in the same package:
  tool-to-module  ← conversion pipeline (convert, assess-only, containers,
                     bioconda-recipe, update-standards)
  validate-module    ← module validation suite (test, fix, review)
Separating them means validate-module can be extracted into a standalone
`nf-module-tools` package later without renaming any user-facing commands.
validate_cli.py imports from validate.py, fix.py, review.py only.
It must NOT import from ingest, discover, assess, infer, container, or generate.

## Functionality Discovery Rules (CRITICAL)
Discovery runs BEFORE assess and infer. It determines how many modules to generate.

Rule-based detection (always run first, fast, no LLM):
  - click/typer @command decorators → one FunctionalitySpec per decorated function
  - argparse add_subparsers + add_parser calls → one per subparser name
  - shell case statement dispatch → one per case branch that calls external tools
  - multiple top-level scripts in repo (each *.py, *.sh, *.R not named utils/helpers/common) →
    one FunctionalitySpec per script file

LLM-based detection (only when rule-based finds zero or one functionality):
  - Send repo manifest + combined code to LLM
  - Ask it to identify distinct independently-invokable operations
  - Each LLM-detected functionality carries DetectionMethod.LLM and lower confidence
  - If LLM also finds only one functionality: proceed as single-module case (no selection UI)

nf-core module count rules:
  - One module per distinct, independently-invokable functionality
  - Never merge two functionalities with different I/O into one module
  - A single monolithic script with no subcommands = one module (do not artificially split)
  - Low-confidence functionalities (< 0.50) shown in UI but not pre-selected

Selection UI:
  - Always shown when ≥ 2 functionalities detected, regardless of confidence
  - Pre-select all functionalities with confidence ≥ 0.70
  - Low-confidence entries shown with ⚠ and not pre-selected
  - --functionalities flag for non-interactive selection (comma-separated names)
  - --all-functionalities flag to select all without UI

## nf-core Standards (CRITICAL)
Always follow nf-core/tools 3.5+ conventions:
- Use TOPIC CHANNELS for versions (not versions.yml file)
- Input channels: `tuple val(meta), path(files)` pattern
- Output channels: must include `versions` topic channel
- Labels: process_single / process_medium / process_high / process_high_memory
- Container: both Docker (quay.io/biocontainers) AND Singularity URLs required
- meta.yml: must include EDAM ontology terms where known
- All params set via ext.args in conf/modules.config, never hardcoded in module

## Container Handling (Two Phases)

### Phase 1 — Discovery (container.discover())
Run ALL checks in parallel, collect every available option as a ContainerOption.
Never stop early. Return a ContainerDiscovery with all options found.
Checks to run: Dockerfile in repo, environment.yml, requirements.txt,
Singularity.def, Bioconda/BioContainers lookup.

### Phase 2 — Selection (container.select())
Present options to the user and return their choice.
Default option priority order (used when user accepts default or --no-interaction):
  1. Dockerfile in repo
  2. BioContainers (if tool is Tier 1–2, i.e. known Bioconda package)
  3. Generate from environment.yml
  4. Generate from requirements.txt
  5. Convert from Singularity.def
  6. Stub

IMPORTANT: For Tier 1–2 tools (known Bioconda entry), BioContainers is ranked ABOVE
repo files in the default order — the community standard image is preferred for
well-known tools. For Tier 3–5 tools, Dockerfile/generated options rank first.

User can override default via:
  - Interactive menu (TTY detected automatically)
  - --container dockerfile|biocontainers|generate|stub flag
  - --no-interaction flag (use default silently, no menu shown)

## Complexity Tiers
- Tier 1: Known tool, single I/O, Bioconda entry exists → full generation
- Tier 2: Known tool, minor ambiguity (multi-output, optional params) → generation with warnings
- Tier 3: Custom script, inferrable I/O, no container → generation + container from options
- Tier 4: Complex custom code, ambiguous channels, multiple tools → partial generation + TODOs
- Tier 5: Cannot proceed → report why, suggest manual approach

## Code Style
- Python 3.10+, type hints required on all functions
- Click for CLI (not argparse)
- Jinja2 for all template rendering
- Never hardcode tool names or container URLs — always infer or look up
- Pydantic v2 for all data models (CodeSource, ModuleSpec, TestSpec, ContainerOption,
  ContainerDiscovery)
- Rich for terminal output (progress bars, status panels, tier badges, selection menus)

## Testing
- pytest for unit tests
- Fixtures in tests/fixtures/
- Mock Anthropic API calls in tests — never make real API calls in CI
- Mock all httpx calls — never hit real APIs in CI
- Test discovery and selection phases independently
- Test Tier-aware default ordering explicitly

## When Adding New Templates
1. Add .j2 template to src/tool_to_module/templates/
2. Add corresponding Pydantic model field if new data needed
3. Add test fixture that exercises the new template
4. Update meta.yml.j2 if new output type added

## Standards Schema (CRITICAL)
Never hardcode nf-core conventions directly in Python code or Jinja2 templates.
Always read them from tool_to_module.standards (the standards/ subpackage),
which loads src/tool_to_module/standards/data/nf_core_standards.json.
Import: from tool_to_module.standards import get_standards

Examples of things that MUST come from the schema, not be hardcoded:
  - Valid process labels (process_single, process_medium, etc.)
  - Required output channel names (versions)
  - Container registry base URLs
  - Required meta.yml fields
  - EDAM ontology term mappings
  - Test data index (nf-core/test-datasets paths and metadata)
  - Generation tool commands and templates

When writing any code that references an nf-core convention, ask:
  "Is this in the schema?" If not, add it to nf_core_standards.json first,
  then read it via get_standards() from tool_to_module.standards.

## Fix Safety Rules (CRITICAL)
The fix command MUST follow these rules without exception:

1. NEVER modify files without showing a diff and receiving explicit human approval.
   Every proposed fix is shown as a coloured Rich diff panel before any write occurs.

2. The source of every proposed fix MUST be visible in the diff panel:
   "[rule] Missing ext.args pattern" vs "[llm] Output pattern corrected to *.sorted.bam"
   Users need to know whether to trust a fix as deterministic or as a suggestion.

3. Rule-based fixes (Class A) ONLY for failures with deterministic structure:
   wrong emit name, missing topic channel tag, container URL format errors,
   missing ext.args, meta.yml field mismatches. Rules never call the LLM.

4. LLM-assisted fixes (Class B) ONLY when reading code in context is required:
   wrong output filename pattern, wrong process label, unclear channel description.
   LLM fixes are clearly labelled and carry a lower trust signal in the diff panel.

5. Class C failures are NEVER fixed automatically. The tool explains what it found,
   why it cannot fix it, and what the human needs to do. Then it stops.

6. After applying fixes, always re-run validation. Never claim "fixed" without verifying.

## Review Scope
The review command performs static analysis only — it does not run tests.
It checks nf-core style conventions: channel naming patterns, label appropriateness
for the inferred compute requirements, ext.args usage, meta.yml completeness,
and EDAM ontology coverage. It produces a ReviewReport with severity levels:
ERROR (blocks submission) | WARNING (should fix) | INFO (suggestion).
Test data selection follows a strict three-strategy priority order.
Never skip a strategy without trying it first:

  Strategy 1 — Match existing nf-core/test-datasets (test_data_match.py)
    Query the test_data.index in the standards schema.
    Match on: format tag, organism preference, paired/single-end, size.
    Always prefer the smallest file that satisfies the channel spec.

  Strategy 2a — Derive from existing nf-core data (test_data_derive.py)
    Only if Strategy 1 finds no match.
    Subset or transform an existing nf-core/test-datasets file (VCF region, FASTA extract,
    BAM subsample). Generates derive_test_data.sh. The output file must be PRed to
    nf-core/test-datasets. Supported derivations in standards.derivation_templates.
    Preferred over chain for most formats — a stored file keeps CI test runtime low.

  Strategy 2b — Chain an upstream nf-core module (test_data_derive.py)
    Some inputs are the natural output of a known upstream nf-core module.
    Emit a setup {} block in main.nf.test — no stored file needed.
    Known chain modules are in standards.chain_modules.
    Use ONLY when: (a) the input file would be too large for nf-core/test-datasets,
    OR (b) the upstream module runs extremely fast (e.g. index generation from a
    small reference). nf-core discourages setup{} blocks as a general default because
    they increase test runtime — derive is preferred when the file is storable.

  Strategy 3 — Nextflow stub mode
    Use when data is too large, proprietary, or cannot be derived from nf-core data.
    Module includes a stub: block; nf-test runs with options "-stub".
    Generate a commented TODO block in main.nf.test explaining what is needed.
    Never silently produce an nf-test that will always fail.

Each ChannelSpec carries a test_data_strategy field set during LLM inference (Prompt 8).
The LLM should classify each input as: "standard_format" | "generatable" | "custom".
test_data_match.py and test_data_derive.py use this hint to route efficiently.

## Testing Protocol

Run tests in escalating stages. Never skip a stage to reach a later one.

### Stage 1 — After each prompt (fast, always)
Run pytest for the modules touched in this prompt only.
Fix any failures before proceeding. Do not run the full suite yet.

### Stage 2 — After a milestone or bug fix (full unit suite)
  pytest -x -q -m "not network and not llm"
Stop on first failure (-x). Identify whether each failure is a regression
(was passing before this change) or pre-existing. Fix regressions before
proceeding; report pre-existing failures without fixing them.

### Stage 3 — After Prompt 22 (architectural integrity)
Run in this order, reporting results separately:
  pytest tests/test_import_boundaries.py -v
  pytest -x -q -m "not network and not llm"
  pytest tests/test_chain.py -v
A boundary violation is a different class of problem from a unit failure —
do not conflate them.

### Stage 4 — Real-world integration (Celltypist)
  tool-to-module convert https://github.com/Teichlab/celltypist \
    --outdir /tmp/celltypist_test \
    --docs https://celltypist.readthedocs.io/en/latest/ \
    --no-interaction
Then verify every item in the Prompt 16 manual checkpoint in
docs/development-tutorial.md. Report each checkpoint item as PASS or FAIL.
Do not summarise — list every item explicitly.

### Stage 5 — Discovery robustness (network, slow)
  pytest tests/test_discovery_robustness.py -v -m network
Report failures with full pytest output, not a summary. If a repo clone
fails due to a transient network error, retry once before reporting.

### Stage 6 — LLM contract tests (opt-in, costs API credits)
  ANTHROPIC_API_KEY=<key> pytest tests/test_llm_contract.py -v -m llm
For each failing invariant, identify whether the failure is in the raw LLM
output (invariant never produced) or in the post-processing guard (invariant
produced but guard failed to enforce it). These require different fixes.

### Failure protocol — apply at every stage
1. Show the full pytest output for the failing test, not a paraphrase.
2. Identify the root cause before proposing a fix.
3. Confirm the fix is general — not a special-case patch for the failing test.
4. Re-run the failing test after the fix to verify it passes.
5. Re-run the full suite for the affected module to check for regressions.
Never mark a task complete while any test in the relevant suite is red.

## Common Pitfalls
- nf-core 3.5+ uses topic channels: `emit: versions, topic: 'versions'`
- Singularity URLs use `https://depot.galaxyproject.org/singularity/` prefix
- Docker URLs use `quay.io/biocontainers/` prefix
- Always quote container strings in main.nf (interpolation issues)
- meta.yml `tools` section requires homepage and documentation URLs
- Generated Dockerfiles must be pushed to a registry before nf-core submission
- Discovery runs ALL checks — do not short-circuit even if Dockerfile is found
- Selection applies Tier-aware default — check tier BEFORE setting is_default flags
- Never hardcode nf-core conventions in code — always read from standards/loader.py
- validate_cli.py must not import from ingest, discover, assess, infer, container,
  or generate — this boundary enables future extraction of the validation suite
- standards/ subpackage must not import from any sibling module in tool_to_module
  (no imports from ingest, assess, container, etc.) — same extraction reason

Tier 1 — High confidence, known tool:

  - func.detection_method is CLICK_DECORATOR, ARGPARSE_SUBPARSER, or SHELL_CASE_STATEMENT

Tier 2 — Good confidence, known tool with complexity:
  - Same as Tier 1 but: multiple output patterns detected, OR optional flags present,
    OR tool in bio.tools API (https://bio.tools/api/tool/{tool}/)
  - OR detection_method is MULTI_SCRIPT_REPO with a known tool
  → confidence: 0.75–0.89

Tier 3 — Moderate confidence, custom script:
  - Python/R/Perl with inferrable I/O (argparse/click/optparse arguments present in
    code_section, OR input/output variable names clearly identifiable)
  - No Bioconda entry found, but language and purpose are clear
  - func.detection_method is any (rule-based or LLM)
  → confidence: 0.50–0.74, add warning: "No container found — stub will be generated"

Tier 4 — Low confidence, complex custom code:
  - code_section calls multiple external tools

--- PART 2: Strategy selection logic ---

plan_derivation() ALWAYS returns a DerivationPlan (never None).
Strategy priority order — use the FIRST that applies:

  1. DERIVE: if any entry in standards.derivation_templates has applicable_tags
     that intersects channel.format_tags, AND channel.test_data_strategy != "custom",
     AND the estimated output file size is <= 50MB.
     Derive is the preferred fallback over chain for most formats. A stored derived
     file keeps CI test runtime low, which is nf-core's stated priority. The derive
     script needs to be run once and the output PRed to nf-core/test-datasets, but
     that cost is paid once; every subsequent CI run is fast.

  2. CHAIN: if any entry in standards.chain_modules has produces_tags that is a
     superset of channel.format_tags, AND either:
     (a) the input file is estimated to be too large for nf-core/test-datasets (>50MB), OR
     (b) the chain_module entry has fast: true in the standards schema (meaning the
         upstream process completes in seconds — e.g. index generation from a small
         reference).
     Chain MUST NOT be used as a general convenience to avoid PRing test data.
     nf-core explicitly discourages setup{} blocks as a default because they increase
     test runtime — two processes run instead of one on every CI invocation.

     What "chain" means in practice: the generated nf-test runs a real upstream
     nf-core module inside a setup {} block to produce the input file on the fly,
     rather than reading it from disk. The canonical nf-core example is
     ABRICATE_SUMMARY, which requires ABRICATE_RUN to produce its report input.
     The setup block feeds a raw genome FASTA from nf-core/test-datasets into
     ABRICATE_RUN, whose report output flows directly into ABRICATE_SUMMARY's
     input channel. No intermediate file is stored. The same pattern applies to
     index files (BWA_INDEX, SAMTOOLS_FAIDX) where storing the index would be
     wasteful but re-generating it from a small FASTA is very fast.
     The trade-off: the test is slower (two processes instead of one) and requires
     the upstream module to be installed and passing in the test environment.

  3. STUB: all other cases, including:
     - channel.test_data_strategy == "custom" or "unknown"
     - format too large for GitHub and no chain module available
     - no matching derivation template and no chain module

Add a fast: bool field to each entry in standards.chain_modules:
  "samtools_faidx": { ..., "fast": true }   ← index from small FASTA: ~1 second

  "samtools_sort":  { ..., "fast": false }  ← sorting a BAM: slower, derive preferred

This field drives condition (b) in the CHAIN check above.

def get_imports(path: Path) -> set[str]:
    """Return all module names imported by the file at path."""
    tree = ast.parse(path.read_text())
    imports = set()
    for node in ast.walk(tree):
        if isinstance(node, ast.Import):
            for alias in node.names:
                imports.add(alias.name)
        elif isinstance(node, ast.ImportFrom):
            if node.module:
                imports.add(node.module)
    return imports

    Troubleshooting

    Discovery found zero functionalities This happens when the source has no detectable CLI structure and the LLM also couldn't identify distinct operations. Common causes: the script is a helper/utility rather than a standalone tool, the file is too short to analyse, or the repo contains only config files. Try tool-to-module assess-only SOURCE to see the Tier assessment, or pass the specific script file path rather than the whole repo directory.

    Discovery found functionalities I don't want Use --functionalities name1,name2 to specify exactly which to generate, or run without --no-interaction to use the selection menu and deselect entries. Low-confidence entries (marked ⚠) are not pre-selected by default.

    LLM detection produced wrong functionality boundaries LLM_INFERENCE fires only when no rule-based detector finds ≥ 2 functionalities. If the results look wrong, the rule-based detector may have missed the entry points: - click/typer: non-standard decorator wrappers can fool the AST detector - bash: the case statement detector requires $1, ${1}, CMD, or COMMAND as the dispatch variable; other patterns are missed Use --functionalities to name the parts you want explicitly, bypassing detection.

    Multiple modules generated but process names conflict Each module gets a process name like TOOLNAME_SUBCOMMAND. Conflicts can occur with LLM-detected functionalities that produce ambiguous names. Solution: run with --functionalities to supply clearer names, or edit the process line in the generated main.nf directly before testing.

    tool-to-module test exits immediately: "nf-test not found" Install nf-test: conda install -c nf-core nf-test or follow https://www.nf-test.com/installation. The test command requires both nf-core (for lint) and nf-test on PATH.

    tool-to-module fix reports "LLM returned unparseable diff" The LLM proposed a fix but the response wasn't a valid unified diff.     Appendix — Lessons Learned and Tool-Specific Knowledge

    tool-to-module has no built-in mechanism for capturing lessons learned during debugging or review sessions. This is intentional for now — the tool already has two channels for feeding context back into generation:

    • --docs — pass any URL or local file, including a hand-written notes file, to give the LLM context it couldn't infer from code alone.
    • --existing-modules — pass previously generated modules to align container URLs, labels, and channel conventions.

    The recommended workaround for tool-specific lessons is a plain markdown file maintained alongside the module:

    modules/liana/TOOL_NOTES.md

    Annotate it after each session with anything that would have helped generation — version string location, non-standard output paths, known lint failures and their fixes, quirks of the conda package. Pass it back in on future runs:

    tool-to-module convert liana-py/ --docs modules/liana/TOOL_NOTES.md

    Recurring generation mistakes (e.g. the LLM consistently using the wrong container registry, or omitting a required meta.yml field) belong in the standards schema rather than in a notes file. Update standards/data/nf_core_standards.json, bump schema_version, and run tool-to-module update-standards. This fixes the mistake for all future modules, not just the next one.

    CLI wrapper over library API — a distinct tool category. Some tools present as CLI tools (they have a console_scripts entry point and accept --indata, --model, --outdir style flags) but are primarily Python libraries whose CLI is a thin dispatch wrapper. Celltypist is an example: its README devotes far more space to celltypist.annotate() than to celltypist --indata. These tools have:

    • One console_scripts entry point dispatching via flags (not subcommands)
    • A rich Python API documented in sections separate from the CLI section
    • File-based I/O in the CLI that maps cleanly to nf-core channels
    • Multiple distinct operational modes expressed as flags rather than subcommands

    The README's CLI section is the right source for identifying which modes exist. For celltypist: --indata --model --outdir (annotate), --indata ... --majority-voting (annotate + voting), --update-models (download), --show-models (list). These are real distinct modes — the LLM-based splitting into 6 sub-functionalities was semantically reasonable, it just used the wrong mechanism (doc interpretation) instead of code signals.

    The correct treatment for pipeline use is two modules: celltypist/annotate and celltypist/train. Utility modes with no file I/O (--update-models, --show-models) are not pipeline steps and should be filtered out during discovery.

    Discovery must be code-signal-driven, not LLM-driven. The LLM must never be called during the discovery phase to interpret, expand, or split functionalities. Splitting must come from source code analysis only, following a three-level strategy:

    1. Explicit subcommands (add_subparsers()): one FunctionalitySpec per add_parser("name") call. Name = subcommand name.
    2. Mutually exclusive flag groups or mode-switching flags: parse the AST for add_mutually_exclusive_group() blocks and flags that trigger fundamentally different execution paths. One spec per mode, named toolname_modename. Filter out modes with no file input and no file output — these are not pipeline steps.
    3. Flat parser, no clear mode split: one spec for the whole tool. Name = console_scripts key.

    The LLM is only involved from infer.py onward, once a FunctionalitySpec with a real code section has been produced by code-signal-based discovery.

    meta must be enforced as a post-processing invariant, not inferred by the LLM. Every nf-core module must accept a meta map as the first input for sample tracking — without it, $meta.id in the tag line and def prefix = task.ext.prefix ?: "${meta.id}" fail at runtime. During the celltypist debugging session, meta was correctly generated in one run and then silently dropped in the next when the optional channel fix changed how infer.py processed the input list. The LLM does not see meta in the tool's source code (it's a Nextflow convention, not a CLI argument) and will therefore omit it whenever it doesn't appear in the prior context.

    The fix is a post-processing step in infer.py that runs after the LLM returns a ModuleSpec and enforces the meta convention unconditionally: - If the first input is not val(meta) or tuple val(meta), path(...), insert it - If meta is present but not first, move it to first position - Never allow meta to be optional or dropped

    If there is one required path input, combine with meta into a tuple: tuple val(meta), path(input_data). If there are multiple required path inputs, keep meta as a standalone val(meta) on its own line. This is a structural invariant — it should never be left to LLM inference.

    Optional CLI arguments must map to optional Nextflow channels. Celltypist's --gene-file and --cell-file are only required when the input is in .mtx format — they are genuinely optional for the common .h5ad and .csv input paths. If infer.py generates these as required channels, the module will fail for any non-mtx input. The fix is path gene_file optional true in main.nf. General rule: any CLI argument marked default=None or guarded by a conditional in the argparse handler should map to an optional channel, not a required one. infer.py should detect this from the AST or from docs context and set optional: true in the channel spec accordingly.

    Stub test data strategy is expected for tools requiring domain-specific inputs. Celltypist requires a .pkl model file and an .h5ad or .csv count matrix — neither is in nf-core test-datasets. All input channels therefore fall back to stub strategy in the test data table, which is correct behaviour. The TODO comments in tests/main.nf.test give guidance for adding real test data later. This is not a bug — it reflects the genuine difficulty of testing single-cell tools in a CI environment. The stub strategy produces a syntactically valid test that passes lint and confirms channel wiring without requiring real biological data.

    Confidence score ambiguity. The word "confidence" and percentage scores currently appear in two unrelated contexts with different meanings:

    • Discovery (→ Auto-selected: celltypist (confidence 78%)) — how strongly the evidence supports this being a genuine CLI entry point. Should be driven by signal type: console_scripts entry → ~95%, argparse block detected → ~75%, public function name only → ~50%.
    • Assessment (Tier 3 Confidence: 74%) — how clearly the code fits the criteria for the assigned tier. A Tier 3 at 74% means "probably Tier 3 but some signals are ambiguous."

    These are different things. A user seeing both assumes they mean the same thing — they don't. Fix before release: rename the assessment score to tier match and display it as a qualitative label (strong / moderate / weak) rather than a percentage, reserving percentages for discovery where they have a clear meaning as a selection threshold.

    Tier descriptions missing from output. A user seeing Tier 3 Confidence: 74% with no context has to read the docs to understand what Tier 3 means and whether 74% is good or bad. The tier descriptions are already defined in the standards schema and in assess.py — they just aren't surfaced in the Rich output. Fix before release: add a one-line description to the assessment box:

    ╭─────────────────────────────────────────────────────╮
│ Tier 3  ·  moderate match                           │
│ Has CLI structure but no Bioconda/PyPI container    │
│ ⚠ No container found — stub will be generated      │
╰─────────────────────────────────────────────────────╯

    The five tiers for reference:

    Tier Meaning
    1 CLI tool, in Bioconda, container ready
    2 CLI tool, in PyPI/conda-forge, container resolvable
    3 CLI tool, container must be built from Dockerfile or generated
    4 Ambiguous structure, limited I/O signals
    5 Cannot auto-generate — manual creation required

    Future extension: If the same tool-specific quirks appear repeatedly across users, a structured tool_knowledge/ directory — written after a successful session, passed in automatically on the next run — would automate what the notes file does manually. Build this only once real-world usage reveals which information is actually worth capturing.

    Perl tools and other non-Python wrappers assess as Tier 5. TrimGalore is a Perl wrapper around cutadapt and FastQC — it has no Python AST to analyse, no importable entry points, and its CLI is defined in a Perl script that the rule-based detectors do not parse. General rule: tools whose primary implementation language is not Python, R, or shell will tend toward Tier 4–5 unless they have a very clean argparse- style CLI. The right response is to report the tier honestly and let the user decide whether to proceed manually — not to attempt generation and produce a broken module. For the regression test suite, prefer Python or R tools with Bioconda packages as reference targets; Perl wrappers are poor regression targets regardless of how widely used they are. cutadapt (Python, single-functionality, Bioconda-backed) is a better third regression target than TrimGalore for this reason.

    Appendix — Library-to-Module (Future Capability)

    Why library tools are out of scope today

    nf-core modules wrap command-line invocations. The Nextflow process script shells out to a tool:

    script:
"""
celltypist \
    --input $input \
    --model $model \
    --output $prefix.csv
"""

    Python libraries whose public API consists of importable functions (decoupler, liana, scanpy) don't have this shape. There is no command to shell out to. Wrapping them requires first generating a CLI wrapper script — a bin/tool.py that parses arguments, calls the library function, and writes output files. That script doesn't exist in the repo; it would need to be created.

    Why library-to-module, not library-to-tool

    The wrapper script is an implementation detail of the module, not a standalone deliverable. nf-core modules can ship a bin/ directory with helper scripts that the process calls — module consumers never see the script directly, they call the process. So the right target is a self-contained module where the bin script is generated inside the module directory as part of the conversion:

    library function → bin/tool_name.py (generated) → nf-core module wrapping it

    Architecture: an extension of tool-to-module, not a separate tool

    The two paths share significant code — everything from assess onward is identical. Both produce a ModuleSpec and feed it into the same Jinja2 templates. The divergence is only in discovery and a new bin script generation step:

    CLI tool:  ingest → discover (CLI entry points) ──────────────────────┐
                                                                       ├→ assess → infer → generate
Library:   ingest → discover (library functions) → bin_gen.py (new) ──┘

    The right implementation is a strategy pattern inside discover.py, not a separate tool or a separate ingest path:

    # discover.py
def discover(source, mode="auto"):
    if mode == "auto":
        mode = "library" if _looks_like_library(source) else "cli"
    if mode == "cli":
        return _discover_cli(source)      # existing logic
    if mode == "library":
        return _discover_library(source)  # __init__.py expansion

    _looks_like_library checks: no argparse/click/shell entry points found AND __init__.py exports functions or submodules rather than a CLI group. The --library-mode flag forces the choice rather than auto-detecting, which is useful when auto-detection gets it wrong.

    The only genuinely new file is bin_gen.py, which takes a FunctionalitySpec with a library function signature and generates the bin/ wrapper script. It is called between discover and assess in the library path only. Everything else — ingest.py, assess.py, infer.py, generate.py, container.py, validate.py — is unchanged and shared.

    Key design decisions when building this

    Which functions to wrap. Not every public function in a library deserves an nf-core module. The selection criteria: does the function take file-like inputs and produce file-like outputs when called from a pipeline context? For decoupler, tl.run_ulm(mat, net) takes an AnnData and a network DataFrame — both naturally represented as files (h5ad, csv). Internal helpers, plotting functions, and format converters are lower priority.

    The __init__.py submodule expansion logic (detect __all__, expand submodules, extract public functions) developed during the decoupler debugging session is the right starting point for _discover_library. Cap the candidate list early (suggest MAX_LIBRARY_FUNCTIONS = 20) to avoid processing hundreds of internal helpers.

    Type-to-format mapping. The hardest inference problem is mapping Python types to file formats. A principled default table:

    Python type Nextflow channel type Default format
    AnnData file .h5ad
    pd.DataFrame file .csv
    np.ndarray file .npy
    str (path) file inferred from name
    int, float, bool val —
    dict file .json

    This table belongs in standards/data/nf_core_standards.json once built, so it can be updated without code changes.

    Version capture. The generated bin script must print the library version to stdout for the nf-core version capture pattern. For pip-installed packages this is always importlib.metadata.version("package_name").

    Testing. Because bin_gen.py is the only new module, it needs its own write-first test prompt (following the same pattern as Prompts 5 and 12). The rest of the test suite requires no changes — assess, infer, generate, and validate tests all run against ModuleSpec regardless of how it was produced.

    Appendix — nf-core Module Regression Tests

    Note: This appendix is now superseded by Prompt 28, which implements regression.py, RegressionScore, parse_module, and semantic_score as production code with full tests. The background and design rationale below remains useful reading before tackling Prompt 28.

    The idea

    Existing nf-core modules are ground truth. There are hundreds of them with known-correct main.nf, meta.yml, and environment.yml. Running tool-to-module convert against the source tool and comparing the output against the reference module gives a reproducible, automatable quality signal that doesn't require human review — and scales across the entire nf-core module catalogue.

    This is the eval-driven development loop that makes LLM-based tools reliable over time: generate → compare → identify gap → fix in infer.py or generate.py or add to lessons-learned → repeat. Each failure mode caught becomes a permanent improvement rather than a one-off manual fix.

    What to compare — semantic diff, not string diff

    The LLM will never reproduce exact formatting, variable names, or comment style of a hand-written module. Exact string diffing will always fail. What you can compare is structure and semantics:

    Dimension What to check
    Process name Matches expected (e.g. SAMTOOLS_SORT)
    Container URL Matches BioContainers reference for the same version
    Input channels Generated names are a subset of reference inputs; types match
    Output channels Emit names match reference (these are the public contract)
    Version capture Pattern present, uses the right command (tool --version)
    Script block No TODO placeholders; tool invocation references correct input channel names
    meta.yml keywords Overlap with reference keywords above a threshold

    Score each dimension independently. A module that gets container, process name, outputs, and version capture right but has slightly different input channel names is much better than a module that fails on all dimensions — and that distinction is lost in a binary pass/fail.

    Implementation sketch

    # tests/test_regression.py

REFERENCE_MODULES = [
    # (tool_github_url, docs_url, nf_core_module_path)
    ("https://github.com/samtools/samtools", None,
     "nf-core/modules/modules/nf-core/samtools/sort"),
    ("https://github.com/alexdobin/STAR", None,
     "nf-core/modules/modules/nf-core/star/align"),
    # add more as the suite grows
]

@pytest.mark.regression
@pytest.mark.parametrize("tool_url,docs_url,reference_path", REFERENCE_MODULES)
def test_against_reference_module(tool_url, docs_url, reference_path, tmp_path):
    run_convert(tool_url, docs_url, outdir=tmp_path)
    generated = parse_module(tmp_path)
    reference = parse_module(reference_path)
    score = semantic_score(generated, reference)
    assert score.process_name    # must match
    assert score.container_url   # must match
    assert score.output_channels >= 0.8   # 80% of reference outputs present
    assert score.no_todos        # no placeholder comments

    parse_module reads main.nf into a structured object (process name, inputs, outputs, container, script body). semantic_score compares two parsed modules dimension by dimension.

    Selecting reference modules

    Not all nf-core modules are equally suitable as regression targets:

    • Prefer recently updated modules — older modules use patterns superseded by current nf-core lint rules. Pin to modules with a last-commit date after the nf-core 2.14 release or that pass nf-core lint cleanly today.
    • Prefer single-functionality tools — samtools/sort, fastqc, trimgalore are cleaner targets than multi-subcommand wrappers like gatk4/haplotypecaller which involve complex conditional logic.
    • Prefer tools with Bioconda packages — these exercise the full BioContainers resolution path, which is the most important container discovery path to keep correct.
    • Avoid modules that wrap scripts or custom containers — these have non-standard patterns that tool-to-module isn't designed to replicate.

    Where this fits in the build order

    Regression tests are not part of the initial 22-prompt tutorial — they require a working implementation to generate output to compare against. The right time to build them is after the Prompt 16 manual checkpoint confirms end-to-end generation is working correctly for Celltypist.

    Add them as an optional Prompt 23 once the core 22 prompts are stable:

    Prompt 23 (optional): Regression test suite
Dependencies: all 22 prompts complete, at least one real-world module
generated and reviewed successfully.
Goal: implement tests/test_regression.py with semantic_score comparator
and 3-5 reference modules from nf-core/modules.
Write-first: define the RegressionScore dataclass and semantic_score
signature before implementing parse_module or the comparator logic.

    Appendix — Test Data from Tool Documentation (Future Capability: Strategy 2c)

    The problem

    Strategies 1 and 2 (match and derive) both depend on nf-core/test-datasets as their source of truth. This works well for standard genomics formats — BAM, VCF, FASTA, FASTQ — because nf-core/test-datasets has comprehensive coverage of these. It breaks down for domain-specific formats that the nf-core community hasn't needed yet:

    • Single-cell formats: .h5ad, .loom, .mtx + barcodes/features
    • Spatial transcriptomics: .h5, Visium directories
    • Mass spectrometry: .mzML, .raw
    • Proteomics: .fasta databases, .pin files
    • Machine learning models: .pkl, .pt, .onnx

    For Celltypist, the workflow correctly falls through to stub because there is no .h5ad or .pkl in nf-core/test-datasets. But Celltypist's own documentation links to a minimal example dataset used in every tutorial. That dataset is small, publicly available, already used by the tool's authors for testing, and exactly the right semantic content for an nf-test. The workflow currently ignores it entirely.

    Strategy 2c: fetch from tool documentation

    Strategy 2c sits between Strategy 2b (chain) and Strategy 3 (stub) in priority order. It triggers when:

    • Strategy 1 (match) finds no nf-core/test-datasets file for the channel's format
    • Strategy 2b (chain) finds no upstream nf-core module that produces this format
    • The LLM identified a suggested_test_data_url for this channel during inference

    If all three conditions hold, the workflow generates a fetch_and_minimise.sh script that downloads the example file and reduces it to a size suitable for CI testing.

    How it fits into the existing architecture

    During infer.py: Add a suggested_test_data_url: str | None field to ChannelSpec. When the LLM is inferring channel specs and --docs content is present, it should look for example dataset links in the documentation and populate this field. The prompt addition to infer.py is small:

    For each input channel, if the documentation mentions an example file or dataset
URL that matches this channel's format, set suggested_test_data_url to that URL.
Only set it if the URL points directly to a downloadable file (not a tutorial page).
If no URL is found, leave it as null.

    During test_data_derive.py: Add a Strategy 2c branch to plan_derivation():

    # Strategy 2c: fetch from tool documentation
if (
    channel.suggested_test_data_url is not None
    and not _is_derivable(channel, standards)   # Strategy 2 already failed
    and not _is_chainable(channel, standards)   # Strategy 2b already failed
):
    return _plan_fetch(channel, standards, tool_name)

    _plan_fetch() returns a DerivationPlan with strategy="fetch" and a new fetch_url field. The source enum gains a FETCHED value alongside DERIVED, CHAINED, and STUB.

    A new Jinja2 template: templates/derive_test_data/fetch_and_minimise.sh.j2 generates a shell script that:

    1. Downloads the file from suggested_test_data_url using curl -L
    2. Runs a format-appropriate minimisation step (see table below)
    3. Writes the minimised file to tests/data/{tool_name}_test.{ext}
    4. Prints PR instructions for nf-core/test-datasets

    Minimisation by format:

    Format Minimisation tool Target size Command sketch
    .h5ad anndata (Python) ~100 cells, ~500 genes adata[:100, :500].write_h5ad(out)
    .loom loompy (Python) ~100 cells loompy.combine([ds[:, :100]], out)
    .mtx scipy.sparse first 100 rows slice + mmwrite
    .pkl none — keep as-is — copy only; models can't be meaningfully subset
    .mzML pyteomics first N spectra parse + rewrite

    For .pkl and other opaque binary formats where minimisation isn't possible, the script downloads the full file with a size warning and a note that it may be too large for nf-core/test-datasets.

    License checking

    Before generating the fetch script, the workflow should inspect the docs page for license terms. The LLM can do this as part of the suggested_test_data_url extraction step:

    If you identify a suggested_test_data_url, also set suggested_test_data_license
to the license of the example dataset if you can determine it from the docs page.
Use "unknown" if you cannot determine it. Use "cc0", "mit", "apache2", "gpl",
or "proprietary" as values where known.

    The fetch script is generated regardless, but includes a prominent comment:

    # LICENSE: {{ license }}
# IMPORTANT: Verify this dataset's license before submitting to nf-core/test-datasets.
# nf-core/test-datasets requires permissive licensing (CC0, CC-BY, or equivalent).
# Ask on Slack #test-data if unsure.

    If license == "proprietary", the script is not generated and the channel falls through to stub with a specific message: "Example data identified but license is incompatible with nf-core/test-datasets. Using stub mode."

    What this would have done for Celltypist

    With Strategy 2c in place, the Celltypist run would have:

    1. During inference, identified https://celltypist.cog.suse.com/models/models.zip (or the readthedocs tutorial dataset link) as suggested_test_data_url for the model channel, and a PBMC example .csv or .h5ad for the input channel
    2. Generated fetch_and_minimise.sh that downloads the PBMC example and subsets it to 100 cells using anndata
    3. Produced a working nf-test referencing tests/data/celltypist_test.h5ad instead of a stub block
    4. Flagged the model .pkl as non-minimisable with a size warning

    The nf-test would still need the user to run fetch_and_minimise.sh and PR the file to nf-core/test-datasets — but it would be a real, runnable test rather than a stub placeholder.

    Where this fits in the build order

    Strategy 2c requires: - Prompt 8 (infer.py) — adds suggested_test_data_url to ChannelSpec - Prompt 11 (test_data_derive.py) — adds the _plan_fetch() branch

    It should be implemented as an extension to Prompt 11 once the core three strategies are stable. It is more complex than Strategies 1–3 because it involves: - A new field in the LLM inference prompt - A new DerivationPlan strategy value - A new Jinja2 template with format-specific minimisation logic - A license check heuristic

    For that reason it is scoped here as a future capability rather than part of the core 28 prompts. The right time to build it is after Prompt 16 (the Celltypist checkpoint) confirms that stub mode is the actual bottleneck for real-world tools.
